# Vibely AI: пользовательские музыкальные векторы

Небольшая нейросеть на PyTorch, которая кодирует музыкальные вкусы пользователя в вектор фиксированного размера. 
Поверх неё поднят HTTP‑сервис на FastAPI.

## Структура

- `model.py` — модель `UserMusicEncoder` (эмбеддинги треков, артистов и жанров).
- `train.py` — пример обучения на синтетических данных, сохранение весов в `user_encoder.pt`.
- `inference.py` — функции для построения вектора пользователя из его истории.
- `nearest_neighbours.py` — поиск ближайших пользователей по косинусной схожести.
- `main.py` — пример скрипта, который считает векторы синтетических пользователей и выводит похожесть.
- `service.py` — FastAPI‑сервис, который отдаёт вектор пользователя и его соседей по вкусу.
- `requirements.txt` — зависимости Python для части `ai`.

## 2. Обучение модели

### 2.1. Быстрый старт (synthetic)

По умолчанию в `train.py` используется синтетический набор пользователей для демонстрации.

Запуск обучения (из папки `ai`):

```bash
python train.py
```

В конце обучения веса модели сохраняются в файл:

- `user_encoder.pt` — рядом с `train.py`.

### 2.2. Обучение на продовых данных

Для обучения на реальных данных используется JSONL‑файл со строками формата:

```json
{
  "user_id": "user_123",
  "track_id": 10,
  "genre_id": 3,
  "artist_ids": [21, 22],
  "rating": 1.0
}
```

Один JSON‑объект на строку. `rating` — уже числовая шкала предпочтения (например, по той же логике, что в `_liked_to_rating` в `service.py`).

Файл можно собрать на бэкенде из кликов/оценок пользователей (см. ниже идею с веб‑формой).

Запуск обучения на таком датасете:

```bash
python train.py --data-path data/user_events.jsonl --model-path user_encoder_prod.pt
```

Если файл не найден или в нём нет ни одного валидного пользователя, `train.py` автоматически вернётся к синтетическому датасету.

Параметры:

- `--data-path` — путь к JSONL с событиями (опционально).
- `--model-path` — куда сохранить веса.
- `--epochs`, `--steps-per-epoch` — гиперпараметры обучения.
- `--num-tracks`, `--num-artists`, `--num-genres` — размеры словарей (должны совпадать с тем, что использует сервис / маппинги idшников).

> В проде обычно периодически (раз в N часов/дней) обновляют JSONL из логов/БД, переобучают модель и выкатывают новые веса (обновляют `AI_MODEL_PATH` у сервиса).

## 3. Как устроены входные данные

Модель кодирует **события прослушивания** вида:

```python
{
    "track_id": int,        # id трека (внутренний индекс 0..NUM_TRACKS-1)
    "genre_id": int,        # id жанра (внутренний индекс 0..NUM_GENRES-1, один жанр на трек)
    "artist_ids": [int],    # список id артистов (внутренние индексы 0..NUM_ARTISTS-1)
    "rating": float,        # степень "нравится": >0 — нравится, <0 — не нравится, 0 — нейтрально
}
```

Во время инференса для пользователя передаётся **история** — список таких событий, и на её основе строится один вектор пользователя.

В HTTP‑сервисе (`service.py`) эти события приходят в более удобном виде, а шкала "нравится" кодируется строками.

### Шкала "насколько нравится трек"

Поле `liked` в запросе может принимать значения:

- `"strong_like"` → `rating = +1.0`
- `"like"` → `rating = +0.7`
- `"neutral"` или `null` → `rating = 0.0`
- `"dislike"` → `rating = -0.7`
- `"strong_dislike"` → `rating = -1.0`

Это преобразование реализовано в функции `_liked_to_rating` внутри `service.py`.

> Чем больше по модулю `rating`, тем сильнее событие влияет на итоговый вектор пользователя. Знак определяет направление (нравится/не нравится).

## 4. Запуск API‑сервиса

Сервис реализован на FastAPI в файле `service.py`. Он один раз загружает веса модели и затем обрабатывает HTTP‑запросы.

### 4.1. Переменные окружения

Необязательные (есть разумные значения по умолчанию):

- `AI_MODEL_PATH` — путь к файлу весов модели (по умолчанию `user_encoder.pt` рядом с `service.py`).
- `AI_DEVICE` — устройство для инференса (`cpu` или, при наличии, `cuda`), по умолчанию `cpu`.
- `AI_NUM_TRACKS` — максимальное количество треков (размер словаря треков). Должен быть ≥ максимального `track_id`, который ты используешь.
- `AI_NUM_ARTISTS` — максимальное количество артистов.
- `AI_NUM_GENRES` — максимальное количество жанров.

По умолчанию в `service.py` заданы достаточно крупные значения, пригодные для большого сервиса:

- `NUM_TRACKS = 500_000`
- `NUM_ARTISTS = 100_000`
- `NUM_GENRES = 64`

Если ты знаешь точные размеры словарей, лучше задать их явно через env.

### 4.2. Запуск через uvicorn

Из корня проекта (так как `ai` — пакет):

```bash
uvicorn ai.service:app --host 0.0.0.0 --port 8000
```

Либо из папки `ai`:

```bash
cd ai
uvicorn service:app --host 0.0.0.0 --port 8000
```

Проверь health‑чек:

```bash
curl http://localhost:8000/health
```

Ожидаемый ответ:

```json
{ "status": "ok" }
```

## 5. Эндпоинты сервиса

### 5.1. Регистрация/обновление пользователя и получение его вектора

`POST /users/register`

Тело запроса:

```json
{
  "user_id": "user_123",
  "tracks": [
    {
      "id": 10,
      "genre_id": 3,
      "artist_ids": [21, 22],
      "liked": "strong_like"
    },
    {
      "id": 11,
      "genre_id": 5,
      "artist_ids": [30],
      "liked": "neutral"
    },
    {
      "id": 12,
      "genre_id": 1,
      "artist_ids": [5, 6],
      "liked": "dislike"
    }
  ]
}
```

- `id` — внутренний индекс трека: `0 <= id < NUM_TRACKS`.
- `genre_id` — внутренний индекс жанра: `0 <= genre_id < NUM_GENRES`.
- `artist_ids` — список внутренних индексов артистов.
- `liked` — строка из шкалы выше (можно не передавать, тогда считается `"neutral"`).

Ответ:

```json
{
  "user_id": "user_123",
  "embedding": [0.123, -0.045, ..., 0.089]
}
```

Эмбеддинг — это нормализованный вектор фиксированной размерности (`user_emb_dim` в `model.py`, по умолчанию 16).

Сервис также сохраняет эмбеддинг пользователя в памяти процесса, чтобы потом можно было быстро искать похожих пользователей.

### 5.2. Получение вектора уже зарегистрированного пользователя

`GET /users/{user_id}/embedding`

Пример:

```bash
curl http://localhost:8000/users/user_123/embedding
```

Ответ:

```json
{
  "user_id": "user_123",
  "embedding": [0.123, -0.045, ..., 0.089]
}
```

Если пользователь ещё не регистрировался в этом процессе — будет 404.

### 5.3. Поиск ближайших пользователей по вкусу

`GET /users/{user_id}/nearest?top_k=10`

Пример:

```bash
curl "http://localhost:8000/users/user_123/nearest?top_k=5"
```

Ответ:

```json
{
  "user_id": "user_123",
  "neighbors": [
    { "user_id": "user_42", "similarity": 0.92 },
    { "user_id": "user_7",  "similarity": 0.88 }
  ]
}
```

Похожие пользователи ищутся по **косинусной схожести** их эмбеддингов. Сам пользователь в список не включается.

> Важно: сейчас эмбеддинги всех пользователей хранятся **в памяти процесса сервиса** (MVP). В проде стоит сохранять их в БД / векторном хранилище (Postgres + pgvector, Redis, Milvus и т.п.).

## 6. Использование модели напрямую (без HTTP)

Если хочешь использовать модель прямо из Python‑кода, можно сделать так:

```python
import torch
from ai.model import UserMusicEncoder
from ai.inference import build_user_embedding

device = "cpu"

model = UserMusicEncoder(
    num_tracks=500_000,
    num_artists=100_000,
    num_genres=64,
).to(device)
model.load_state_dict(torch.load("ai/user_encoder.pt", map_location=device))
model.eval()

user_history = [
    {"track_id": 10, "genre_id": 3, "artist_ids": [21, 22], "rating": 1.0},
    {"track_id": 11, "genre_id": 5, "artist_ids": [30], "rating": 0.0},
]

embedding = build_user_embedding(model, user_history, device=device)
print(embedding.shape)  # (user_emb_dim,)
```

## 8. Идея веб‑формы для сбора продовых данных

Простой рабочий сценарий:

1. Пользователь заходит на веб‑страницу и загружает/подтягивает свой плейлист.
2. Фронтенд показывает треки из плейлиста с кнопками "нравится / не нравится / пропустить".
3. Каждое нажатие отправляет на бэкенд событие вида:

   ```json
   {
     "user_id": "user_123",
     "track_external_id": "123456",
     "artist_external_ids": ["a1", "a2"],
     "genre_external_id": "g3",
     "liked": "like"
   }
   ```

4. Бэкенд:
   - маппит внешние ID в внутренние индексы (через таблицы `tracks`, `artists`, `genres`),
   - переводит `liked` в числовой `rating`,
   - пишет событие либо в таблицу `user_events`, либо сразу в JSONL `data/user_events.jsonl` в формате, который ожидает `train.py`.

5. Отдельный офлайн‑скрипт (или cron‑задача) периодически запускает:

   ```bash
   python ai/train.py --data-path data/user_events.jsonl --model-path ai/user_encoder_prod.pt
   ```

6. После успешного обучения обновляется переменная окружения `AI_MODEL_PATH` у FastAPI‑сервиса, и он начинает использовать новые веса.

Дополнительно можно:

- после первичной оценки плейлиста предлагать пользователю **случайные/подобранные треки** для оценки (бэкенд выбирает кандидатов из каталога и рендерит ту же форму с like/dislike), а события продолжают писаться в тот же поток данных для обучения.
